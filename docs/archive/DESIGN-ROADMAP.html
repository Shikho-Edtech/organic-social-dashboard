Roadmap from design audit · 2026-04-18

Design Roadmap

26 improvements from the audit, each with implication + risk, grouped into three execution batches. Companion doc to DESIGN-AUDIT.html.

The audit was an honest read — the dashboard is structurally sound and statistically correct; what's missing is the polish layer. Everything below is prioritized by impact per hour. Batch 1 is the "one day, baseline lift" pass. Batch 2 is the craft pass that moves the product from competent to opinionated. Batch 3 is the strategic rebuild — ship incrementally.

Items already handled in LEARNINGS/DECISIONS (mobile nav dropdown, popup clamping, data-driven YAxis, tap-toggle tooltips, etc.) are listed at the bottom and excluded from batches.

Batches at a glance

Baseline quality lift

~1 day · all Small effort

WCAG AA compliance on the common paths, loading/error states, brand palette finally shows up, Strategy verdict hero gets the weight it deserves, Plan gets a "today" marker. No IA changes, no new components, low risk.

Batch 1 — Foundation

The one-day pass that visibly lifts quality everywhere

14 items · all S effort · low risk · safe to ship as one commit
#1WCAG AA contrast fix: text-slate-400 → text-slate-500CriticalSLow risk
globals.css + ~80 occurrences across components + Recharts #94a3b8 → #64748b

ImplicationMeta labels, sample-size badges, render timestamps, week-ending dates become legible on mobile in daylight. 2.85:1 → 4.58:1. Reserve slate-400 for ornamental · separators only. Side-effect: secondary info becomes slightly less visually "distant" from primary — intentional, since it was unreadable before.

#2Global focus-visible ring for buttons / links / summariesCriticalSLow risk
app/globals.css — single base-layer rule

ImplicationKeyboard users can tab through every page without losing position. WCAG 2.4.7 pass. Visual side-effect: brief indigo ring appears on any click until blur — some teams find this "noisy" on a mouse-first app; mitigated by using focus-visible (not focus) so only keyboard interactions trigger it.

#3app/error.tsx + app/loading.tsx at rootMajorSLow risk
app/error.tsx, app/loading.tsx — Next.js auto-wires them

ImplicationCold loads stop looking broken. Skeleton KPI strip + skeleton chart block for 300–1500ms instead of blank white. "Application error" screens replaced with a friendly card + "Try again" button. Low risk, but skeleton design matters — keep it simple (pulsing rounded blocks) so it doesn't feel fake.

#4Nav desktop active-tab color → brand-shikho-indigoMinorSLow risk
components/Nav.tsx line ~81

ImplicationDesktop matches mobile (which already uses indigo). Brand shows up in the most-used UI element. One-line change.

#5Palette reorder: indigo + pink first in BarChart / DonutMajorSLow risk
components/BarChart.tsx PALETTE, components/Donut.tsx

ImplicationPrimary data color across the app becomes brand-aligned. The dashboard starts feeling like Shikho instead of generic shadcn. No chart has a hardcoded color-to-meaning interpretation (nobody is reading "cyan = pillars"), so reordering is safe.

#6Reels retention curve: bars → lineMajorSLow risk
app/reels/page.tsx line ~336 — use TrendChart variant="line"

ImplicationCanonical viz for a continuous function. The classic cliff-then-decay shape becomes legible at a glance instead of hidden in 60 skinny bars. Uses existing TrendChart — no new component.

#7Engagement Breakdown: 6-slice donut → horizontal barMajorSLow risk
app/engagement/page.tsx line ~214

ImplicationCleveland & McGill — angle comparison fails past 3–4 slices; bar positions on a common scale are more accurate for the viewer. Same data, same colors, easier to read.

#8Remove redundant kind text badge from ChartCard headerMinorSLow risk
components/Card.tsx line ~51

ImplicationLeft-border color already communicates kind (observed / ai / derived). Removing the duplicate text badge declutters the chart header where title + (i) + sample-size + kind were competing for one row. Document kinds once in DataFooter.

#9Strategy Weekly Verdict hero redesignMajorSMed risk
app/strategy/page.tsx line ~128

ImplicationThe most important card on the dashboard finally looks important. Bigger type (text-2xl lg:text-4xl), stronger gradient, explicit "Read full verdict →" button replacing the tiny 10px "more" label. Med risk because of mobile height — pair with M4 fix (line-clamp-1 on headline below sm:) so it doesn't push funnel charts off-screen.

#10Plan: auto-detect + highlight "today" day-cardMinorSLow risk
app/plan/page.tsx

ImplicationCalendar says "you are here" immediately. Today's day-card gets an indigo ring + opens by default. Watch out: compute today in Asia/Dhaka (BDT), not server UTC — the pipeline generates for BDT and server time will be UTC on Vercel.

U6Standardize disclosure chevrons across Plan + StrategyMinorSLow risk
app/plan/page.tsx, app/strategy/page.tsx

ImplicationCurrently three patterns: Plan uses >→v, Strategy findings use v→^, Strategy verdict uses >. Pick one — down-chevron rotating 180° on open — and apply everywhere <details> is used.

M1Explore filter tap targets ≥ 44×44 on mobileMajorSLow risk
app/explore/ExploreClient.tsx filter toolbar

ImplicationWCAG 2.5.5 pass. Currently 4 filter pills compressed into a 375px row produce mis-taps between Pillar and Format. Stack vertically below sm: with py-2.5 text-sm. Desktop keeps compact.

M5Plan slot dividers visible: divide-slate-100 → divide-slate-200PolishSLow risk
app/plan/page.tsx slot list

ImplicationCurrent 1.03:1 dividers are invisible. 3.8:1 dividers separate slots cleanly. Or alternatively odd:bg-slate-50/40 for a subtle zebra. CSS-only.

P3Reels: cache retention-curve parse (3000 → 50 parses)PolishSLow risk
app/reels/page.tsx retention-curve aggregation

ImplicationOuter 60-second loop × 50 reels × parseRetentionCurve = 3000 JSON parses. Parse once per reel, cache in local array, iterate seconds inside. CPU win on the Reels page, no visible change.

Batch 2 — Craft pass

The pass that makes the dashboard feel designed

13 items · mostly M effort · medium risk · 2–3 days, multiple commits
#11Drop colorByIndex where color conveys nothingMinorMMed risk
Pillar / Format / Hook / Spotlight charts across pages

ImplicationWhen a bar chart's value is already on the x-axis, coloring each bar differently is chart junk. Default to single brand color for value-comparison charts. Keep colorByIndex only where the color maps to a category referenced elsewhere. M risk because every chart needs a per-use review.

#12Suspense boundaries around chart gridsMinorMLow risk
app/layout.tsx + each page

ImplicationKPI strip renders first (fast), chart grid streams in with skeleton (slower). Better perceived performance on slow mobile. Pairs with #3. Requires splitting each page's data-fetch into a KPI-only path and a charts path.

#14Reels table → mobile card-listMajorMLow risk
app/reels/page.tsx line ~385

Implication9-column table is invisible below md:. Swap to card list (mirror Explore Top 10 pattern, already in codebase) that stacks all metrics per reel. Table still shown at md:+. Resolves M3 from the audit.

#16"Rendered {ts}" → "Data as of {ts}" from actual fetch timeMinorMLow risk
components/PageHeader.tsx + lib/sheets.ts

ImplicationCurrently shows request arrival time — conflicts with 5-min cache. New field tracks actual Sheets fetch time. Also plumb it to Plan (currently hides the timestamp via showPicker={false} — Plan is exactly where staleness matters most, see U4). M effort because lib/sheets.ts needs to propagate the timestamp.

#17Introduce <EmptyChart> componentMinorMLow risk
new component + ChartCard prop

ImplicationTiming already does good empty states. Overview / Trends / Engagement leave Recharts blank (reads as buggy). One component, wired via ChartCard prop, replaces every ad-hoc empty-state div. Consistent "try this" messaging.

#18"Best X" KPI colors → category-semantic (locked)MinorSLow risk
app/engagement/page.tsx Best X strip

ImplicationColor stops being decoration. Best Format=Reel pulls the pink Reels use on Plan; Best Pillar pulls its pillar-canonical color; Best Hook / Spotlight likewise. Adds one lookup function mapping "value → canonical color." Aligns with the project's own CLAUDE.md rule that color should convey meaning.

V2Typography: drop text-[10px] and text-[9px] entirelyMinorMLow risk
all pages — grep for text-\[10px\] / text-\[9px\]

ImplicationFive small sizes (xs / 11 / 10 / 9 / sm) compress into unreadable noise. Three sizes instead: text-xs (12) for labels, text-[11px] for eyebrow/uppercase, nothing below 11. If a label doesn't fit at 11px it's too verbose — shorten instead of shrink. M effort because it's an audit of every usage.

V4Standardize inline SVG stroke + sizePolishSLow risk
all pages — icon SVGs

ImplicationIcons stop looking hand-drawn. Two strokes: stroke-2 ornamental (chevron, info), stroke-2.5 affordance (close, check). Three sizes: 12 / 14 / 16. Icons read as one system.

V5Consistent spacing rhythm (space-y-6 at wrapper)MinorSLow risk
each page's top-level wrapper

ImplicationKPI rows using mb-6, chart grids mb-4, some with none — gaps inconsistent as you scroll. Wrapper uses space-y-6; strip per-grid margin-bottoms. Feels tidier.

V6Cards get a visual family: shadow + hover + KPI gradientPolishMMed risk
components/Card.tsx + every card usage

Implicationshadow-sm baseline (was flat), hover:shadow-md hover:-translate-y-px on clickable cards only, optional bg-gradient-to-br from-white to-slate-50/40 on KPI cards. Stops feeling like generic Tailwind. M risk: over-doing the gradient/shadow lands in "skeuomorphic" territory — restraint.

Pg-OvOverview: replace Engagement Mix donut with "Biggest movers" calloutMinorMLow risk
app/page.tsx

ImplicationEngagement Mix donut is redundant with Engagement page's Engagement Breakdown. Replace with a one-line so-what ("Shares ↑ 43% WoW · comments flat · reactions ↓ 8%"). Higher information density on the most-visited page.

Pg-EnEngagement: Recommendations card synthesizing the four Best XMinorMLow risk
app/engagement/page.tsx top of page

ImplicationFour Best X KPIs require the reader to stitch the narrative. A one-line recommendation up top ("Post more Reels with Teacher spotlight — both beat others by >40%") does the stitching for them. Synthesis derived from the same data; no new pipeline work.

Pg-TrTrends: weekly-snapshot small-multiples cardMinorMLow risk
app/trends/page.tsx

ImplicationCurrently four loosely-grouped weekly charts (volume, reach, ER, shares). One dense card with four small multiples reads as "the weekly snapshot" and consolidates scrolling. Trends stays as its own page (decision: keep for social media team distribution).

Pg-LgLogin: data-flow illustration replacing cadence bulletsPolishMLow risk
app/login/page.tsx

ImplicationFour cadence bullets (daily / Monday / source / cache) replaced with a visual data-flow — Meta API → Google Sheets → Dashboard, with cadence + freshness indicators on each arrow. Inline SVG so it scales crisply. The login screen stops being text-in-a-box and becomes the first aesthetic statement the product makes. Fits the brand palette (indigo → pink → orange gradient on the flow line).

Batch 3 — Strategic

The IA + structural rebuild — ship incrementally, not in one commit

4 items · L effort · higher risk · separate week of focused work
#13Timing: slot × day bar pair → day × hour heatmapMajorMMed risk
app/timing/page.tsx (new heatmap component)

ImplicationCanonical "when to post" viz. One chart answers the question instead of two bar charts the user has to mentally cross-reference. Medium build (new component, Recharts doesn't have a native heatmap — either custom SVG grid or svg + div hybrid). Color scale choice matters: sequential (low→high reach-weighted ER), probably the same brand-indigo ramp used on KPIs. Also fixes the audit's "6 time slots miss 0–5am" bucketing issue by switching to actual hours.

Pg-ExExplore rebuilt as a filter-first workbenchMajorMMed risk
app/explore/ExploreClient.tsx

ImplicationFilter toolbar becomes sticky at the top (position: sticky, stays visible while you scroll the results). KPI strip pushes down + shrinks — it's summary data already on Overview. Top 10 post list becomes the prominent "results" section — bigger cards, likely pagination (25/50/100). Charts sit between filters and results as at-a-glance indicators of what the current filter slice looks like. Mental model: Overview = dashboard (passive summary), Explore = workbench (drill into a specific question).

#19Consistent page template across all routesMinorLHigh risk
every page

ImplicationTemplate: header → KPI strip (max 5) → primary chart (full or 2-up) → secondary charts → detail. Every page conforms. Cognitive load drops as users learn one pattern. L effort because each page gets a structural review and reshuffle. High risk during transition — temporarily breaks familiarity. Ship page-by-page, not all at once.

#20Full accessibility sweepMinorLLow risk
InfoTooltip, all popovers, all chart wrappers

ImplicationInfoTooltip gets aria-describedby + Escape-to-close. All popovers (DateRangePicker, MultiSelect, Group-by) get Escape handling + aria-expanded / aria-haspopup. Each chart gets a "View data" button revealing a keyboard-accessible table (Recharts tooltips are hover-only, unavoidable without this). Takes the dashboard from "partial WCAG AA" to "near-full." Functionally low risk.

Decisions locked in

✓ Q1 · Best X colors = category-semantic
Best Format=Reel pulls pink, Best Pillar pulls its pillar color, etc. Color becomes informative instead of decorative. (Batch 2, #18)
✓ Q2 · Plan → Strategy merge = defer
Keep /plan separate through Batch 3. Revisit after page-template work lands — at that point the merge is mechanical. Removed from Batch 3 scope.
✓ Q3 · Keep Trends as its own page
Dashboard hasn't been distributed to the social media team yet — if Trends is useful to them post-distribution, keep it. Decision: do not merge Trends into Overview. Trends stays as a standalone route.
✓ Q4 · Login illustration = build it
Replace cadence bullets with a data-flow illustration (Meta API → Google Sheets → Dashboard). Added to Batch 2 as Pg-Lg. Goal: login feels aesthetic, not text-in-a-box.
✓ Q5 · Explore = rebuild as proper workbench
Added to Batch 3 as Pg-Ex. Sticky filter toolbar, KPIs de-emphasized, Top 10 becomes the prominent results section with pagination. Explore's purpose becomes "answer a specific drill question," distinct from Overview's passive summary role.

Already handled — excluded from batches

Items resolved in prior work and documented in LEARNINGS.md / DECISIONS.md / CLAUDE.md: flex-wrap → flex-col sm:flex-row (DECISIONS 2026-04-18) · popup viewport clamp (DECISIONS 2026-04-18) · data-driven YAxis width (DECISIONS 2026-04-18) · tap-toggle InfoTooltip (DECISIONS 2026-04-18) · labelled dropdown nav on mobile · KpiCard break-words + responsive sizes · Plan slot 3-col flex stack on mobile · adaptive MIN_N gating · Strategy date-range clarity · CI lower-bound Best X ranking · Day 2O staleness banners on Strategy + Plan (just shipped).

Recommendation: ship Batch 1 as a single commit in one focused session. Low-risk, S-effort across the board, and baseline quality visibly jumps. Batch 2 gets broken into 3–4 smaller commits by theme (streaming; color system; empty states; page-level callouts) so each can be reviewed in isolation. Batch 3 is a separate week of focused work — the Timing heatmap alone is worth a day.

Net effort estimate: Batch 1 ~1 day · Batch 2 ~2–3 days · Batch 3 ~5–6 days (Explore workbench added). Roughly 9 working days to close the full audit. Every batch is independently shippable, so you can stop at any boundary and the dashboard is still materially better.
